QA Engineering

Automation Testing Guide

A practical reference for API, UI, load, and mobile testing — covering the tools, languages, and patterns used across the full testing spectrum, from exploratory sessions to fully automated CI pipelines.

Testing strategy

Effective QA is layered. Exploratory testing discovers unknown risks early. Automated checks — from contract tests to full E2E flows — then catch regressions on every commit and keep releases safe at speed.

Exploratory Integration End-to-End Load

The testing pyramid

Invest most in fast, isolated unit tests. Build a solid API and integration layer. Keep E2E tests targeted at critical user journeys — they are the most expensive to write and maintain.

Many unit tests Some API tests Few E2E tests
Unit / Component Fast & isolated

Validate individual functions. No external calls. Millisecond feedback in the developer's editor.

Integration / API Contract testing

Verify service interactions, HTTP contracts, and data shapes at the API layer without a browser.

End-to-End Full user journeys

Drive real browsers through critical flows — login, navigation, form submission — across the full stack.

Load & Stress Testing

Load testing verifies behaviour under expected traffic volumes. Stress testing pushes beyond normal limits to find the system's breaking point and measure its recovery. Both are essential before major releases and infrastructure changes.

Apache JMeter

Open Source
Back end Java JavaScript Groovy

The industry standard for load and performance testing. Supports HTTP, HTTPS, FTP, JDBC, SOAP, REST, and more through a rich plugin ecosystem. Test plans are built in a GUI and run headlessly in CI pipelines with HTML dashboard output.

Run JMeter headless via CLI
jmeter -n -t test_plan.jmx \
       -l results.jtl \
       -e -o ./html-report/

k6

Open Source · Cloud
Back end JavaScript

Developer-first load testing with a JavaScript scripting API. Lightweight, version-control friendly, and native to CI/CD. Grafana k6 Cloud adds distributed execution, real-time charts, and threshold alerting.

k6 — basic load script
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = { vus: 50, duration: '30s' };

export default function () {
  const res = http.get('https://api.oscarrondon.com/v1/content', {
    headers: { Authorization: 'Bearer YOUR_KEY' },
  });
  check(res, { 'status 200': (r) => r.status === 200 });
  sleep(1);
}

Locust

Open Source
Back end Python

Python-native distributed load testing. User behaviour is defined as Python classes, making complex multi-step workflows easy to script. A built-in real-time web UI provides live stats and user-count controls without extra tooling.

Locust — user class definition
from locust import HttpUser, task, between

class PortfolioUser(HttpUser):
    wait_time = between(1, 3)

    @task(3)
    def list_content(self):
        self.client.get("/v1/content",
            headers={"Authorization": "Bearer YOUR_KEY"})

    @task(1)
    def get_single_item(self):
        self.client.get("/v1/content/cnt_1024",
            headers={"Authorization": "Bearer YOUR_KEY"})

SoapUI / ReadyAPI

Free · Enterprise
Back end JavaScript Groovy

Dual-purpose tool combining functional API testing with a built-in load runner. ReadyAPI (enterprise) adds cloud execution, real-time reporting, and data-driven load scenarios. Groovy scripts handle dynamic tokens, custom assertions, and multi-step workflows.

Karate (Load)

Open Source
Back end · Front end Gherkin

Unique in that the same BDD .feature files used for functional API testing are executed as load scenarios via karate-gatling — no separate load scripts needed. Eliminates duplication between test types and keeps maintenance low.

API & Back-end Automation

API tests validate endpoints, contracts, data integrity, and error handling in isolation — faster and more reliable than UI tests because they bypass the browser entirely. These run on every commit and form the backbone of a solid regression suite.

Postman / Newman

Free · Pro
Back end JavaScript Python

The most widely-used API testing platform. Collections organise requests by resource or workflow; environments inject variables per stage. Newman CLI runs Postman collections in CI pipelines without a GUI and outputs JUnit XML for test reporting.

Postman — pm.test assertions
pm.test("Status is 200", () => {
  pm.response.to.have.status(200);
});

pm.test("Data array is present and not empty", () => {
  const json = pm.response.json();
  pm.expect(json.data).to.be.an('array').that.is.not.empty;
});

pm.test("Pagination fields are present", () => {
  const { pagination } = pm.response.json();
  pm.expect(pagination).to.have.keys(['limit', 'offset', 'has_more']);
});

REST Assured

Open Source
Back end Java

Java DSL that makes REST API validation read like natural language. Pairs with TestNG or JUnit in Maven/Gradle projects. GPath and Hamcrest matchers enable precise JSON and XML assertions on deeply nested responses.

REST Assured — given / when / then
given()
    .header("Authorization", "Bearer " + API_KEY)
    .queryParam("type", "blog")
    .queryParam("status", "published")
.when()
    .get("/v1/content")
.then()
    .statusCode(200)
    .body("data",           not(empty()))
    .body("data[0].type",   equalTo("blog"))
    .body("pagination.has_more", isA(Boolean.class));

TestNG

Open Source
Back end · Front end Java

JVM testing framework with parallel execution, @DataProvider for data-driven scenarios, and XML-based suite configuration. Used as the runner for REST Assured API tests and Selenium UI tests in the same Java project.

TestNG — data-driven test
@DataProvider(name = "contentTypes")
public Object[][] types() {
    return new Object[][] {
        {"blog"}, {"manual"}, {"whitepaper"}
    };
}

@Test(dataProvider = "contentTypes")
public void testListByType(String type) {
    given()
        .header("Authorization", "Bearer " + API_KEY)
        .queryParam("type", type)
    .when()
        .get("/v1/content")
    .then()
        .statusCode(200)
        .body("data[0].type", equalTo(type));
}

Karate (API)

Open Source
Back end Gherkin

Write API tests in plain Gherkin without Java step definitions. Built-in HTTP client, JSON path assertions, schema validation, and parallel execution. Supports REST, GraphQL, and gRPC — and the same files double as load tests via Gatling.

Karate — feature file
Feature: Content API

  Background:
    * url 'https://api.oscarrondon.com/v1'
    * header Authorization = 'Bearer ' + apiKey

  Scenario: List published blog posts
    Given path 'content'
    And  param type   = 'blog'
    And  param status = 'published'
    When method GET
    Then status 200
    And  match response.data[*].type == '#[] blog'
    And  match response.pagination   == { limit: '#number', offset: '#number', has_more: '#boolean' }

SoapUI (Functional)

Free · ReadyAPI
Back end JavaScript Groovy

Industry favourite for SOAP and REST contract testing. Groovy scripts customise assertions, manage dynamic auth tokens, and chain multi-step flows. Mock services simulate dependencies without live infrastructure — essential for isolated early-development testing.

UI & End-to-End Testing

E2E tests drive real browsers through user-facing flows — login, navigation, form submission, and data display — catching presentation-layer regressions that API tests alone cannot surface.

Selenium

Open Source
Front end Python · Java · JS · C#

The most established browser automation library. WebDriver protocol communicates directly with Chrome, Firefox, Edge, and Safari drivers. Language-agnostic, widely supported in CI, and the foundation for Katalon Studio and other high-level platforms.

Selenium — Python with explicit wait
from selenium import webdriver
from selenium.webdriver.common.by import By
from selenium.webdriver.support.ui import WebDriverWait
from selenium.webdriver.support import expected_conditions as EC

driver = webdriver.Chrome()
driver.get("https://oscarrondon.com/api-docs")

wait = WebDriverWait(driver, 10)
h1 = wait.until(EC.presence_of_element_located((By.TAG_NAME, "h1")))
assert "Getting Started" in h1.text
driver.quit()

Cypress

Open Source · Cloud
Front end JavaScript

Modern E2E framework running inside the browser for real-time execution. Automatic waiting eliminates most flakiness. Time-travel debugging, built-in screenshot and video capture, and a component testing mode all ship out of the box.

Cypress — E2E spec
describe('API Docs', () => {
  beforeEach(() => cy.visit('/api-docs/getting-started'));

  it('shows the page heading', () => {
    cy.get('h1').should('contain', 'Getting Started');
  });

  it('displays the base URL in the code block', () => {
    cy.get('.code-block code')
      .should('contain', 'api.oscarrondon.com/v1');
  });
});

Playwright

Open Source
Front end · Back end Python · JS · Java · .NET

Microsoft's cross-browser automation library covering Chromium, Firefox, and WebKit in a single API. Network interception and request mocking make it equally powerful for API testing. Built-in parallel execution and a trace viewer speed up failure diagnosis.

Playwright — Python example
from playwright.sync_api import sync_playwright

with sync_playwright() as p:
    browser = p.chromium.launch()
    page = browser.new_page()
    page.goto("https://oscarrondon.com/api-docs")
    page.wait_for_selector("h1")
    assert "Getting Started" in page.inner_text("h1")
    browser.close()

UFT One (Micro Focus)

Enterprise
Front end VBScript

Enterprise-grade GUI automation for web, Windows desktop (WinForms, WPF), SAP, and Citrix environments. AI-powered object recognition reduces script maintenance when UIs change. Common in large regulated industries — banking, insurance, and healthcare.

Mobile Testing

Mobile automation tests run against native, hybrid, and mobile-web apps on simulators, emulators, or real devices connected via a cloud device farm such as BrowserStack or Sauce Labs.

Appium

Open Source
Back end · Mobile Java · JS · Python · Groovy

The cross-platform standard for mobile app automation. An Appium server bridges the WebDriver protocol to iOS XCUITest and Android UIAutomator2 drivers without requiring app modifications. Supports native, hybrid, and mobile-web apps in a single framework.

Appium — Python setup
from appium import webdriver

desired_caps = {
    "platformName":   "Android",
    "deviceName":     "emulator-5554",
    "appPackage":     "com.example.app",
    "appActivity":    ".MainActivity",
    "automationName": "UIAutomator2"
}

driver = webdriver.Remote("http://localhost:4723/wd/hub", desired_caps)
driver.find_element("id", "com.example.app:id/login_button").click()

Katalon Studio

Free · Enterprise
Back end · Mobile Java · Groovy

All-in-one automation platform built on Selenium and Appium. A no-code recorder lets manual testers get started quickly; full Groovy scripting gives senior engineers complete control. Covers web, API, mobile, and desktop testing in a single project with built-in CI integration.

BDD with Cucumber & Gherkin

Behaviour-Driven Development shifts the conversation from "what tests cover" to "what the system should do." Scenarios written in Gherkin become living documentation that non-technical stakeholders can read and directly validate.

What is Gherkin?

Gherkin is a structured plain-text DSL using five keywords — Given, When, Then, And, But — to describe observable system behaviour. Files use the .feature extension, are version-controlled alongside application code, and keep requirements and tests permanently in sync.

Gherkin — feature file with Scenario Outline
content.feature
Feature: Content list endpoint

  Background:
    Given the base URL is "https://api.oscarrondon.com/v1"
    And   I authenticate with a valid API key

  Scenario: Retrieve published blog posts
    When  I send GET "/content" with params "type=blog&status=published"
    Then  the response status is 200
    And   the response includes a non-empty "data" array
    And   every item in "data" has status "published"

  Scenario Outline: Filter content by type
    When  I request content with type "<type>"
    Then  all returned items have type "<type>"

    Examples:
      | type       |
      | blog       |
      | manual     |
      | whitepaper |

Cucumber

The most widely-used BDD runner. Available for Java, JavaScript, Ruby, and Python. Step definitions map Gherkin phrases to automation code. Integrates with TestNG, JUnit, Playwright, and Selenium to drive full-stack BDD pipelines from a shared feature file.

Karate

Extends Gherkin specifically for API and load testing without requiring separate step definitions. The DSL includes a built-in HTTP client, JSON path assertions, and schema matching — making a single .feature file a fully self-contained runnable test.

Languages & Frameworks

Choosing the right language for your automation stack aligns test code with your team's existing skills, CI tooling, and the application's primary language.

Python · (P) Versatile & concise

Primary for Locust, Playwright, and Selenium. Ecosystem includes pytest, requests, and Behave for BDD.

Groovy · (G) JVM scripting

Default for Katalon and SoapUI scripts. Concise Java-compatible syntax with Spock Framework support for expressive, readable tests.

JavaScript · (JS) Browser-native

Powers Cypress, k6, and Postman scripts. Node.js enables CLI runners — Newman, Jest, Mocha — in any CI pipeline.

Java · (J) Enterprise standard

Used by REST Assured, TestNG, Appium, and JMeter. Maven and Gradle manage dependencies and parallel execution across large suites.

Gherkin · (Gb)

A structured English dialect — not a programming language. Used by Cucumber and Karate to write executable specifications that product owners, QA engineers, and developers share. Keeps requirements and tests permanently aligned across the team.

VBScript · (VBS)

The scripting engine behind UFT One. Used in legacy enterprise environments for Windows desktop, SAP, and Citrix automation. Keyword-driven and action-based test models are the typical patterns in these stacks.

Best Practices

Consistent automation patterns keep test suites fast, reliable, and cost-effective to maintain as systems and teams grow.

Follow the testing pyramid

Write many fast unit tests, a solid middle layer of API tests, and only as many E2E tests as truly necessary. An inverted pyramid — mostly slow browser tests — creates brittle suites that break CI and burn team confidence in automation.

Isolate test data

Every test should create, use, and clean up its own data. Never rely on state left by another test — shared data causes intermittent failures that are difficult to reproduce and expensive to diagnose across a team.

Gate every pull request with tests

Integrate your suite into GitHub Actions, Jenkins, or GitLab CI. Block merges on test failures. Fast pipeline feedback catches regressions before they reach staging or production and keeps the main branch green.

Tag and categorise tests

Use tags — @smoke, @regression, @load — to run targeted subsets. Smoke tests on each deploy; full regression on a schedule; load tests against staging environments only.

Troubleshooting

Common automation failures and the fastest path to resolving them.

Common Issues

Flaky Tests Intermittent failures

Common causes: missing waits, timing race conditions, shared mutable test data, or unstable environments.

  • Use framework-native waits — Cypress auto-waits; add WebDriverWait in Selenium
  • Never share state between tests — each test owns its full data lifecycle
  • Quarantine flaky tests in a separate suite until the root cause is fixed
  • Re-run failing tests once automatically; flag those that only pass on retry
401 in CI Auth failures in the pipeline

Cause: API keys or tokens not correctly injected into the CI environment.

  • Store secrets as CI environment variables (GitHub Secrets, Jenkins Credentials, Vault)
  • Confirm the variable name in CI matches exactly what the test script reads
  • Use test-environment keys (sk_test_ prefix) to avoid affecting live data
  • Print the first few characters of the resolved key at test startup to verify injection
